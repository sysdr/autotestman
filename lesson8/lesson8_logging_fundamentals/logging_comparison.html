Problems:

  • No timestamps - when did this run?
  • No severity levels - which errors are critical?
  • Output vanishes in CI/CD pipelines
  • Cannot filter or search logs
  • No file-based persistence
  • No context (module, function, line number)

‚úÖ Production Pattern: logging Module

14:32:15 | INFO | Loading test data from: data/test_config.json
14:32:15 | INFO | Successfully loaded 3 test cases
14:32:15 | INFO | [1/3] Executing: Test_HomePage_Load
14:32:15 | INFO | ‚úì Test_HomePage_Load - PASSED
14:32:15 | ERROR | Test 'Test_Missing_Field' missing required fields: ['expected_status']
14:32:15 | WARNING | ‚úó Test_Missing_Field - FAILED
14:32:15 | INFO | Results: 2 passed, 1 failed, 3 total

Benefits:

  • Structured output with timestamps
  • Color-coded severity levels
  • Automatic file persistence (logs/ directory)
  • Module/function/line tracking
  • Filterable by level (DEBUG ‚Üí CRITICAL)
  • CI/CD integration via exit codes

üéØ Why This Matters in Production

The Failure Scenario:

Your test suite runs at 2 AM in CI/CD. A test fails. With print(), you see:

ERROR: Connection failed

With logging, you see:

2024-12-15 02:13:47 | APIClient | ERROR | execute_request:145 | Connection timeout after 5.0s to https://api.prod.company.com/v2/users
2024-12-15 02:13:47 | APIClient | ERROR | Traceback: requests.exceptions.Timeout: HTTPSConnectionPool...

Production Metrics:

Timestamp Resolution: Millisecond Log Rotation: Automatic Searchability: Full-text Integration: ELK/Splunk Ready

Best Practices Demonstrated:

  • Dual Output: INFO+ to console, DEBUG to file
  • Contextual Info: Module, function, line number automatically tracked
  • Exception Handling: logger.exception() captures full traceback
  • Hierarchical Loggers: Per-module loggers with inheritance
  • Formatted Output: Consistent structure across entire framework
